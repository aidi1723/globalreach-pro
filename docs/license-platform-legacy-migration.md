# License Platform Legacy Migration

## Goal

Move older activation systems and product-specific license stores into the unified multi-product license platform.

This is useful for:

- the previous plugin authorization system
- older desktop tool license lists
- manually issued legacy activation keys

## Migration Strategy

1. Assign a stable `product_code` for the legacy system.
2. Import legacy license data into `license_keys`.
3. Keep old clients running temporarily if needed.
4. Move new issuance to the unified platform.
5. Cut old systems over product by product.

## Recommended Product Codes

- Email tool: `globalreach_pro`
- Plugin system: `plugin_system`
- Future scraper tools: `lead_scraper`

## Supported Import Fields

The import script accepts CSV or JSON.

Expected fields:

- `license_key`
- `customer_email`
- `customer_name`
- `status`
- `plan_name`
- `max_activations`
- `expires_at`
- `notes`

Minimum required fields:

- `license_key`
- `customer_email`

## Import Command

From the repository root:

```bash
./.venv/bin/python license-platform/tools/import_legacy_licenses.py \
  --file /path/to/legacy-licenses.csv \
  --product-code plugin_system \
  --product-name "Plugin System" \
  --operator-id 1
```

## Migration Notes

- Imported licenses are recorded with the audit event `license_imported`.
- Existing duplicate `license_key` values are rejected to avoid silent overwrite.
- Importing old keys first allows customer support to keep honoring existing purchases while new keys are generated by the unified backend.

## Recommended Rollout

1. Import historical plugin licenses.
2. Verify counts in the admin console.
3. Stop issuing new licenses from the old system.
4. Point the next client version to the unified license API.
5. Keep the old system read-only until all active users are migrated.
